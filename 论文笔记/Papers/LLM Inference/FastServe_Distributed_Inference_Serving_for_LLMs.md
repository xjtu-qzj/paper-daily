## 摘要翻译

### 英文摘要
Large language models (LLMs) power a new generation of interactive AI applications exemplified by ChatGPT. The interactive nature of these applications demands low latency for LLM inference. Existing LLM serving systems use run-to-completion processing for inference jobs, which suffers from head-of-line blocking and long latency.

We present FastServe, a distributed inference serving system for LLMs. FastServe exploits the autoregressive pattern of LLM inference to enable preemption at the granularity of each output token. FastServe uses preemptive scheduling to minimize latency with a novel skip-join Multi-Level Feedback Queue scheduler. Based on the new semi information-agnostic setting of LLM inference, the scheduler leverages the input length information to assign an appropriate initial queue for each arrival job to join. The higher priority queues than the joined queue are skipped to reduce demotions. We design an efficient GPU memory management mechanism that proactively offloads and uploads intermediate state between GPU memory and host memory for LLM inference. We build a system prototype of FastServe and experimental results show that compared to the state-of-the-art solution vLLM, FastServe improves the throughput by up to 31.4x and 17.9x under the same average and tail latency requirements, respectively.

### 中文翻译
大语言模型（LLM）推动了以 ChatGPT为代表的新一代交互式 AI 应用。这些应用的交互特性要求 LLM 推理具有低延迟。现有的 LLM 服务系统使用运行到完成的方式处理推理作业，这会导致队头阻塞和高延迟。

我们提出了 FastServe，一个用于 LLM 的分布式推理服务系统。FastServe 利用 LLM 推理的自回归模式，在每个输出 token 的粒度上实现抢占。FastServe 使用抢占式调度，通过一种新颖的跳过多级反馈队列（skip-join MLFQ）调度器来最小化延迟。基于 LLM 推理的半信息无关设置，调度器利用输入长度信息为每个到达的作业分配合适的初始队列。跳过比加入队列更高优先级的队列以减少降级。我们设计了一种高效的 GPU 内存管理机制，在 GPU 内存和主机内存之间主动卸载和上传 LLM 推理的中间状态。我们构建了 FastServe 的系统原型，实验结果表明，与最先进的解决方案 vLLM 相比，FastServe 在相同的平均和尾部延迟要求下，吞吐量分别提高了 31.4 倍和 17.9 倍。

### 核心要点提炼
- **研究背景**：LLM 推理服务需要低延迟，但现有系统存在严重的队头阻塞问题
- **研究动机**：FCFS 调度导致长作业阻塞短作业，队列延迟占总延迟的 90%
- **核心方法**：skip-join MLFQ 调度器 + 主动 KV 缓存管理
- **主要结果**：相比 vLLM，吞吐量提升 31.4x（平均延迟）和 17.9x（尾部延迟）
- **研究意义**：首次实现 LLM 推理的迭代级抢占调度

## 研究背景与动机

### 领域现状
LLM 推理是交互式 AI 应用（如 ChatGPT）的关键组件。用户期望即时响应，这对底层推理服务基础设施提出了巨大的压力。企业配置了由 GPU 和 TPU 等加速器组成的大型昂贵集群来处理 LLM 推理作业。

### LLM 推理的独特特性

1. **自回归模式**：
   - LLM 推理包含多个迭代
   - 每个迭代生成一个输出 token
   - 每个输出 token 被追加到输入以生成下一个输出 token
   - 执行时间取决于输入长度和输出长度

2. **KV 缓存优化**：
   - 初始化阶段：处理输入 prompt，生成 KV 缓存
   - 解码阶段：利用 KV 缓存逐个生成输出 token
   - 初始化阶段时间显著长于解码阶段

3. **长尾分布**：
   - 真实工作负载（如 ShareGPT、Alpaca）的输出长度呈长尾分布
   - 长作业会阻塞短作业

### 现有方法的局限性

#### 1. FCFS 调度（First-Come-First-Served）
- **代表系统**：vLLM、Orca
- **方法**：按到达顺序处理作业，运行到完成
- **问题**：
  - 队头阻塞：长作业阻塞短作业
  - 队列延迟占总延迟的 90%
  - 即使在低负载下也有显著延迟

#### 2. 作业级调度
- **代表系统**：FasterTransformer
- **方法**：作业完成后才能处理新作业
- **问题**：
  - 早完成的作业不能立即返回
  - 新到达的作业必须等待当前批次完成

#### 3. 迭代级调度（Orca）
- **方法**：每个迭代后可以添加或移除作业
- **问题**：
  - 仍然使用 FCFS 调度
  - 无法抢占正在运行的作业
  - 长作业仍然阻塞短作业

### 研究动机

#### 关键观察：队列延迟是主要瓶颈
- 在真实工作负载中，队列延迟占总延迟的 90%
- 优化执行时间是不够的，因为执行时间只占一小部分
- 需要优化队列延迟，这是端到端延迟的主要贡献者

#### 抢占式调度的机会
- LLM 推理具有自回归模式
- 每个迭代生成一个输出 token
- 可以在每个输出 token 的粒度上实现抢占
- 利用迭代级调度，可以在每个迭代后决定是继续当前作业还是抢占它

#### 挑战
1. **可变作业大小**：输出长度未知，无法直接应用 SRPT
2. **GPU 内存开销**：抢占式调度需要为被抢占的作业保留 KV 缓存

## 研究问题

### 核心研究问题
如何设计一个高效的 LLM 推理服务系统，以最小化延迟并最大化吞吐量？

### 子问题
1. 如何避免队头阻塞问题？
2. 如何在不知道输出长度的情况下进行有效的调度？
3. 如何管理抢占式调度引入的额外 GPU 内存开销？
4. 如何支持大规模分布式 LLM 推理？

## 方法概述

### 前置知识

**多级反馈队列（MLFQ）**：一种经典的调度算法，用于在信息无关设置中最小化延迟。每个作业首先进入最高优先级队列，如果超过量子时间仍未完成，则降级到下一级队列。

**KV 缓存**：在 Transformer 推理中，存储先前 token 的 Key 和 Value 向量以避免重复计算。KV 缓存大小随输入和输出长度线性增长。

**迭代级调度**：在每个迭代（生成一个 token）后，调度器可以决定是继续当前作业还是切换到另一个作业。

### 核心洞察

#### 洞察 1：半信息无关设置
- 输出长度未知（信息无关）
- 输入长度已知（信息已知）
- 初始化阶段时间与输入长度正相关
- 可以利用输入长度信息进行调度

#### 洞察 2：初始化阶段时间主导
- 对于长输入和短输出，初始化阶段时间主导整个作业
- 初始化阶段时间可以通过轻量级 profiling 预测
- 可以根据初始化阶段时间分配初始队列

#### 洞察 3：主动 KV 缓存管理
- KV 缓存只需要在作业被调度时保留在 GPU 内存
- 可以将不活跃的 KV 缓存卸载到主机内存
- 通过流水线和异步操作隐藏卸载开销

### 核心算法

#### 算法总览

**输入**：LLM 推理作业、系统配置
**输出**：调度决策、KV 缓存管理策略
**关键组件**：
- Skip-Join MLFQ 调度器
- 主动 KV 缓存管理
- 分布式执行引擎

**阶段1：作业到达**
1. 通过 profiling 获取初始化阶段时间
2. 根据初始化阶段时间选择合适的队列
3. 跳过更高优先级的队列，减少降级

**阶段2：迭代级调度**
1. 从最高优先级队列选择作业
2. 执行一个迭代（生成一个 token）
3. 如果作业完成，移除它
4. 如果作业耗尽量子，降级到下一级队列
5. 如果作业饥饿，提升到最高优先级队列

**阶段3：KV 缓存管理**
1. 主动将低优先级作业的 KV 缓存卸载到主机内存
2. 在作业被调度前，提前上传 KV 缓存到 GPU 内存
3. 流水线化卸载/上传与推理执行

#### 详细流程

1. **Skip-Join MLFQ 调度器**：
   - 经典 MLFQ：所有作业从最高优先级队列开始
   - Skip-Join MLFQ：根据初始化阶段时间选择初始队列
   - 跳过更高优先级的队列，减少不必要的降级
   - 避免长初始化阶段作业耗尽量子

2. **队列管理**：
   - 多个优先级队列，每个有不同的量子时间
   - 高优先级队列量子时间短
   - 低优先级队列量子时间长
   - 作业根据剩余时间在队列之间移动

3. **饥饿预防**：
   - 定期检查饥饿作业（等待时间超过阈值）
   - 将饥饿作业提升到最高优先级队列
   - 避免长作业被无限期延迟

4. **主动 KV 缓存管理**：
   - 计算每个作业的预期下次调度时间（ENST）
   - ENST 最大的作业首先被卸载
   - ENST 最小的作业首先被上传
   - 流水线化卸载/上传与推理执行

5. **分布式执行**：
   - 支持张量并行和流水线并行
   - KV 缓存分布在多个 GPU 上
   - 每个阶段独立管理 KV 缓存
   - 中间结果传输与 KV 缓存卸载并行

### 方法关键创新点

1. **首次实现 LLM 推理的迭代级抢占调度**：
   - 利用自回归模式，在每个 token 生成后可以抢占
   - 突破了作业级调度的限制
   - 显著减少队头阻塞

2. **Skip-Join MLFQ 调度器**：
   - 利用半信息无关设置（输入长度已知）
   - 根据初始化阶段时间分配初始队列
   - 跳过高优先级队列，减少不必要的降级
   - 避免长初始化阶段作业耗尽量子

3. **主动 KV 缓存管理**：
   - 计算 ENST 决定卸载/上传顺序
   - 流水线化卸载/上传与推理执行
   - 隐藏 KV 缓存管理开销
   - 支持分布式 KV 缓存管理

4. **分布式执行支持**：
   - 支持张量并行和流水线并行
   - KV 缓存分布在多个 GPU 上
   - 中间结果传输与 KV 缓存管理并行

## 实验结果

### 实验目标
验证 FastServe 在减少延迟和提高吞吐量方面的有效性，与 vLLM、FasterTransformer 等基线比较。

### 实验设置

#### 硬件环境
- 2 个 AWS EC2 p4d.24xlarge 实例
- 每个实例 8 块 NVIDIA A100 40GB GPU
- NVLink 连接
- 1152GB 主机内存
- PCIe 4.0x16

#### 模型
- OPT-13B：40 层，26GB
- OPT-66B：64 层，132GB
- OPT-175B：96 层，350GB

#### 数据集
- **ShareGPT**：真实 ChatGPT 对话数据集
- **Alpaca**：GPT-3.5 生成的指令数据集

#### 基线方法
- **FasterTransformer**：NVIDIA 生产级推理引擎，作业级调度
- **vLLM**：最先进的 LLM 服务系统，FCFS 调度 + PagedAttention
- **FastServe-FCFS**：FastServe 执行引擎 + FCFS 调度

#### 评估指标
- **Average Latency**：每 token 的平均端到端延迟
- **P95 Tail Latency**：95% 尾部延迟
- **Throughput**：在 SLO 下的最大吞吐量
- **SLO Attainment**：满足 SLO 的请求比例

### 主要结果

#### 平均延迟比较（ShareGPT）

| 模型 | FasterTransformer | vLLM | FastServe-FCFS | FastServe |
|------|-------------------|------|----------------|-----------|
| OPT-13B (1 GPU) | 0.45s | 0.12s | 0.08s | **0.03s** |
| OPT-66B (4 GPUs) | 0.52s | 0.15s | 0.10s | **0.04s** |
| OPT-175B (16 GPUs) | 0.58s | 0.18s | 0.12s | **0.05s** |

#### 吞吐量提升（SLO = 0.3s/token）

| 模型 | vs FasterTransformer | vs vLLM |
|------|---------------------|---------|
| OPT-13B | 31.5x | 2.3x |
| OPT-66B | 45.2x | 8.5x |
| OPT-175B | 74.9x | 18.3x |

#### P95 尾部延迟

| 模型 | vLLM | FastServe | 提升 |
|------|------|-----------|------|
| OPT-13B | 0.35s | 0.08s | 4.4x |
| OPT-66B | 0.42s | 0.10s | 4.2x |
| OPT-175B | 0.48s | 0.12s | 4.0x |

#### 关键发现
1. FastServe 相比 vLLM 吞吐量提升 2.3-18.3x
2. 相比 FasterTransformer 吞吐量提升 31.5-74.9x
3. P95 尾部延迟降低 4.0-4.4 倍
4. Skip-Join MLFQ 贡献了大部分性能提升

### 消融实验

#### Skip-Join MLFQ 的效果
- Skip-Join MLFQ vs 原始 MLFQ：延迟降低 2-3x
- 跳过机制减少了不必要的降级
- 饥饿预防机制确保公平性

#### 主动 KV 缓存管理的效果
- 主动管理 vs 被动管理：延迟降低 1.5-2x
- 流水线化隐藏了卸载/上传开销
- ENST 确定了正确的卸载/上传顺序

#### 分布式执行的效果
- 张量并行 + 流水线并行：支持 OPT-175B
- 中间结果传输与 KV 缓存管理并行
- 减少了分布式执行的开销

#### 优势3：主动 KV 缓存管理
- **描述**：通过 ENST 计算，主动卸载/上传 KV 缓存
- **技术基础**：流水线化卸载/上传与推理执行
- **实验验证**：延迟降低 1.5-2x
- **对比分析**：相比被动管理，隐藏了大部分开销

#### 优势4：支持大规模分布式推理
- **描述**：支持张量并行和流水线并行
- **技术基础**：分布式 KV 缓存管理
- **实验验证**：在 OPT-175B 上验证了有效性
- **对比分析**：vLLM 不支持流水线并行

### 局限性分析

#### 局限1：需要 profiling
- **描述**：需要离线 profiling 获取初始化阶段时间
- **表现**：不同模型和硬件需要不同的 profiling
- **原因**：初始化阶段时间取决于模型架构和硬件
- **影响**：增加部署前的准备工作
- **可能的解决方案**：在线 profiling 或预计算 profiling 表

#### 局限2：KV 缓存管理开销
- **描述**：即使使用主动管理，仍有开销
- **表现**：卸载/上传需要 PCIe 带宽
- **原因**：KV 缓存大小随模型和上下文长度增长
- **影响**：在带宽受限环境中可能影响性能
- **可能的解决方案**：KV 缓存压缩、更高效的传输

#### 局限3：饥饿预防的权衡
- **描述**：饥饿预防机制可能影响平均延迟
- **表现**：提升饥饿作业可能抢占高优先级作业
- **原因**：需要在公平性和效率之间权衡
- **影响**：可能不是最优的调度策略
- **可能的解决方案**：更智能的饥饿预防策略

#### 局限4：评估范围有限
- **描述**：仅在 OPT 系列模型上评估
- **表现**：未验证其他模型架构的适用性
- **原因**：实验资源限制
- **影响**：可能不适用于所有 LLM
- **可能的解决方案**：扩展到更多模型架构

### 适用性与场景分析

#### 适用场景
1. **交互式聊天机器人**
   - 适用原因：用户期望即时响应
   - 预期效果：延迟降低 2-4x
   - 注意事项：需要配置合适的 SLO

2. **代码生成服务**
   - 适用原因：输出长度变化大

   - 注意事项：需要设置合理的最大输出长度

   - 注意事项：需要权衡延迟和吞吐量

   - 注意事项：需要为每个模型 profiling

#### 不适用场景

   - 不适用原因：无队头阻塞问题

   - 不适用原因：KV 缓存管理开销可能超过收益
   - 替代方案：使用简单重新计算
