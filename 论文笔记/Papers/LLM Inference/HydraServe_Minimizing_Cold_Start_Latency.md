# HydraServe: Minimizing Cold Start Latency for Serverless LLM Serving in Public Clouds

## 摘要翻译

### 英文摘要
With the proliferation of large language model (LLM) variants, developers are turning to serverless computing for cost-efficient LLM deployment. However, public cloud providers often struggle to provide performance guarantees for serverless LLM serving due to significant cold start latency caused by substantial model sizes and complex runtime dependencies. To address this problem, we present HydraServe, a serverless LLM serving system designed to minimize cold start latency in public clouds. HydraServe proactively distributes models across servers to quickly fetch them, and overlaps cold-start stages within workers to reduce startup latency. Additionally, HydraServe strategically places workers across GPUs to avoid network contention among cold-start instances. To minimize resource consumption during cold starts, HydraServe further introduces pipeline consolidation that can merge groups of workers into individual serving endpoints. Our comprehensive evaluations under diverse settings demonstrate that HydraServe reduces the cold start latency by 1.7-4.7x and improves service level objective attainment by 1.43-1.74x compared to baselines.

### 中文翻译
随着大语言模型（LLM）变体的激增，开发者正在转向无服务器计算以实现成本效益的 LLM 部署。然而，由于巨大的模型大小和复杂的运行时依赖导致的显著冷启动延迟，公共云提供商往往难以为无服务器 LLM 服务提供性能保证。为了解决这个问题，我们提出了 HydraServe，一个旨在最小化公共云中冷启动延迟的无服务器 LLM 服务系统。HydraServe 主动将模型分发到服务器以快速获取，并在工作器内重叠冷启动阶段以减少启动延迟。此外，HydraServe 策略性地在 GPU 之间放置工作器以避免冷启动实例之间的网络争用。为了在冷启动期间最小化资源消耗，HydraServe 进一步引入了流水线合并，可以将工作组合并为单个服务端点。我们在多种设置下的综合评估表明，与基线相比，HydraServe 将冷启动延迟降低了 1.7-4.7 倍，SLO 达成率提高了 1.43-1.74 倍。

### 核心要点提炼
- **研究背景**：无服务器 LLM 服务面临严重的冷启动延迟（>40秒），主要由模型获取和运行时初始化导致
- **研究动机**：公共云中网络带宽受限，现有缓存方法对长尾模型效果有限
- **核心方法**：三级层次化优化（集群级资源分配 + 工作器级重叠 + 推理级流水线合并）
- **主要结果**：冷启动延迟降低 1.7-4.7 倍，SLO 达成率提高 1.43-1.74 倍
- **研究意义**：首个系统性解决无服务器 LLM 冷启动问题的工作

## 研究背景与动机

### 领域现状
LLM 已成为日常工作的核心组件，推动了聊天机器人、编程助手、AI 搜索引擎等广泛应用。随着超过 100 万个模型变体的出现（Hugging Face 记录），无服务器 LLM 服务成为成本效益部署的首选方案，特别适用于使用频率低的长尾模型。

### LLM 冷启动的独特挑战

#### 冷启动阶段分解
在公共云环境中，一个冷启动需要经过以下阶段：

| 阶段 | 耗时 | 主要瓶颈 |
|------|------|----------|
| 容器创建 | 8.52s | 大镜像（8.31GB） |
| 库加载 | 2.65s | Python 运行时、PyTorch、vLLM 等 |
| CUDA 上下文初始化 | 1.56s | GPU 运行时准备 |
| 模型获取 | 24.5s | **主要瓶颈**：受限网络带宽 |
| 模型加载 | 6.87s | GPU 内存传输 |
| 推理（首 token） | 0.6s | 模型执行 |
| **总计** | **>40s** | |

#### 与传统无服务器工作负载的区别
1. **大规模外部状态**：LLM 需要从远程存储按需获取模型权重
2. **复杂运行时依赖**：CUDA、PyTorch、vLLM 等多个专用库
3. **公共云隔离**：容器隔离使运行时无法预先准备

### 现有方法的局限性

#### 1. 内存/SSD 缓存
- **方法**：将频繁访问的模型缓存在内存或 SSD 中
- **问题**：对长尾模型效果有限，这些模型正是无服务器的最大受益者

#### 2. 从活跃工作器获取
- **方法**：从持有相同模型的活跃工作器检索模型权重
- **问题**：需要高带宽网络，对无服务器集群不具成本效益

#### 3. 网络带宽约束
- **经济原因**：云提供商为降低成本选择受限网络的实例
- **实例配置**：AWS EC2 L40S 实例分析显示，增加网络资源可使成本增加 20%-300%
- **关键洞察**：网络升级成本高但仅在冷启动时使用，利用率低

### 研究动机

#### 关键观察：两个优化机会

1. **跨服务器带宽聚合**：
   - 单个服务器带宽有限
   - 将工作器分布在多个服务器上可以聚合带宽
   - 利用 LLM 的分层结构实现流水线并行

2. **冷启动阶段重叠**：
   - 模型获取和运行时初始化具有弱执行依赖
   - 可以并行化而非顺序执行
   - 在工作器级别实现细粒度重叠

## 研究问题

### 核心研究问题
如何在公共云的网络带宽约束下，最小化无服务器 LLM 服务的冷启动延迟？

### 子问题
1. 如何在受限带宽下加速模型获取？
2. 如何重叠冷启动的各个阶段？
3. 如何避免冷启动实例之间的网络争用？
4. 如何在冷启动后合并工作组以恢复最优性能？

## 方法概述

### 前置知识

**流水线并行（Pipeline Parallelism）**：将模型层分布在多个工作器上，中间结果在工作器之间顺序传输。已广泛用于分布式训练和推理。

**无服务器 LLM 服务**：云平台根据实际请求负载自动配置 GPU，用户仅在请求处理期间付费。支持自动扩缩容至零。

**冷启动**：当传入请求找不到可用工作器时，需要创建新工作器的过程。

### 核心洞察

#### 洞察 1：跨服务器带宽聚合
- 单个服务器带宽受限（如 AWS 实例 20-400 Gbps）
- 流水线并行可以将模型获取分布到多个服务器
- 每个工作器只需获取模型的一部分

#### 洞察 2：细粒度阶段重叠
- 模型获取可以与容器创建重叠
- 模型加载（GPU 密集）可以与库加载（CPU 密集）并行
- 获取和加载可以按张量粒度流水线化

#### 洞察 3：流水线合并
- 流水线并行在冷启动后可能影响性能（GPU 共享）
- 工作器可以在后台加载其他工作器的层
- 最终合并为独立工作器，恢复最优性能

### 核心算法

#### 算法总览

**输入**：模型大小、集群资源、用户 SLO
**输出**：流水线并行度、工作器放置策略、GPU 内存分配
**关键组件**：
- 集群级控制器：资源分配 + 网络争用感知放置
- 工作器级优化：模型预取 + 阶段重叠
- 推理级优化：流水线合并

**阶段1：集群级资源分配**
1. 枚举所有部署选择（流水线并行度 1-4）
2. 预测 TTFT 和最坏情况 TPOT
3. 选择满足 SLO 的最小资源消耗方案
4. 网络争用感知的工作器放置

**阶段2：工作器级重叠**
1. 模型预取器：在容器创建前开始获取模型
2. 优先初始化 CUDA 上下文
3. 模型加载与库加载并行执行
4. 按张量粒度流水线化获取和加载

**阶段3：推理级流水线合并**
1. 工作器在后台加载其他工作器的层
2. 迁移运行时状态（KV 缓存）
3. 合并为独立工作器
4. 独立决定是否释放资源或加载完整模型

#### 详细流程

1. **资源分配算法**：
   - TTFT 预测：tc + M/s × max_ratio + tp × (s - w + w/s) + tn × s
   - TPOT 预测：td × (s - w + w/s) + tn × s
   - 枚举所有 (s, w) 组合，选择满足 SLO 的最小资源方案
   - s：流水线并行度，w：全内存工作器数

2. **网络争用感知放置**：
   - 记录每个工作器的获取截止时间和待获取模型大小
   - 检查添加新工作器是否导致 SLO 违规
   - 条件：Si ≤ B/(N+1) × (Di - T)
   - 带宽变化时更新待获取模型大小

3. **模型预取**：
   - 每个 GPU 服务器运行模型预取器
   - 从远程存储预取模型到共享内存
   - 工作器从共享内存流式读取
   - 使用 SafeTensors 格式支持流式访问

4. **阶段重叠**：
   - 优先初始化 CUDA 上下文
   - 模型加载与库加载并行（GPU vs CPU 密集）
   - 按张量粒度流水线化获取和加载

5. **流水线合并**：
   - 冷启动期间使用流水线并行快速响应
   - 后台加载完整模型
   - 迁移 KV 缓存到合并后的工作器
   - 独立决定扩缩容

## 实验结果

### 实验目标
验证 HydraServe 在减少冷启动延迟和提高 SLO 达成率方面的有效性。

### 实验设置

#### 硬件环境
- 4 个 GPU 服务器，每个配备 NVIDIA A10 GPU
- 每服务器 188GB 内存
- 网络带宽：16 Gbps

#### 模型
- OPT-6.7B
- Llama2-7B
- Falcon-7B

#### 基线方法
- **标准冷启动**：顺序执行所有阶段
- **流水线并行**：仅使用流水线并行
- **流水线并行 + 工作器级优化**

#### 评估指标
- **冷启动延迟**：从请求到达到首 token 生成的时间
- **TTFT**：Time-to-First-Token
- **TPOT**：Time-Per-Output-Token
- **SLO 达成率**：满足 SLO 的请求比例

### 主要结果

#### 冷启动延迟比较

| 方法 | 冷启动延迟 | 相比基线 |
|------|-----------|----------|
| 标准冷启动 | ~40s | 基线 |
| 流水线并行 (PP=2) | ~25s | 1.6x |
| 流水线并行 (PP=4) | ~18s | 2.2x |
| HydraServe (PP=4) | ~8.5s | **4.7x** |

#### TTFT 和 TPOT

| 流水线并行度 | TTFT | TPOT |
|-------------|------|------|
| 1 | 12s | 5ms |
| 2 | 7s | 8ms |
| 4 | 4s | 15ms |

#### 关键发现
1. HydraServe 将冷启动延迟降低 1.7-4.7 倍
2. SLO 达成率提高 1.43-1.74 倍
3. 流水线并行对 TPOT 影响较小（中间结果仅 8KB/token）
4. 生产环境评估确认平均延迟降低 2.6 倍

### 流水线并行的权衡分析

#### TTFT 改善
- 流水线并行度 1→2：TTFT 显著降低
- 流水线并行度 2→4：边际改善递减
- 原因：其他冷启动阶段（容器创建、运行时初始化）也有开销

#### TPOT 影响
- 流水线并行对 TPOT 影响较小
- 原因：中间结果传输量小（Llama2-7B 仅 8KB/token）
- 但 GPU 共享可能影响性能

#### GPU 共享问题
- 流水线并行后每个工作器内存需求减少
- 可能导致多个模型共享 GPU
- GPU 计算资源按预留内存比例分配
- 解决方案：流水线合并恢复独立工作器

### 研究价值评估

#### 理论贡献
1. **首次系统性分析无服务器 LLM 冷启动**：

   - 识别模型获取为主要瓶颈
   - 分析公共云网络带宽约束的经济原因

2. **提出三级层次化优化**：

   - 工作器级：模型预取 + 阶段重叠
   - 推理级：流水线合并

3. **设计资源分配算法**：

   - 枚举搜索最优部署方案
   - 平衡性能和资源消耗

#### 实际应用价值
1. **无服务器 LLM 服务提供商**：
   - 降低冷启动延迟，改善用户体验
   - 提高 SLO 达成率
   - 支持更多长尾模型

2. **LLM 应用开发者**：
   - 更快的模型部署和扩展
   - 降低无服务器部署的延迟惩罚
   - 支持更灵活的扩缩容策略

3. **云服务提供商**：
   - 在成本受限实例上提供更好的性能
   - 无需升级网络即可改善冷启动
   - 支持更多模型变体

#### 领域影响
- **短期**：为无服务器 LLM 服务提供实用的冷启动优化
- **中期**：推动无服务器 LLM 服务的普及
- **长期**：可能改变 LLM 服务的部署架构

### 方法优势详解

#### 优势1：显著减少冷启动延迟
- **描述**：通过三级层次化优化，将冷启动延迟从 40s 降至 8.5s
- **技术基础**：流水线并行 + 阶段重叠 + 预取
- **实验验证**：1.7-4.7 倍延迟降低
- **对比分析**：现有方法仅关注模型获取，忽略了阶段重叠的机会

#### 优势2：保持推理性能
- **描述**：流水线合并确保冷启动后恢复最优性能
- **技术基础**：后台模型加载 + KV 缓存迁移
- **实验验证**：TPOT 影响可忽略
- **对比分析**：简单的流水线并行可能因 GPU 共享影响性能

#### 优势3：网络争用感知
- **描述**：避免冷启动实例之间的网络争用
- **技术基础**：带宽分配检查 + SLO 违规检测
- **实验验证**：确保 SLO 达成率
- **对比分析**：现有方法未考虑网络争用

#### 优势4：支持弹性扩缩容
- **描述**：单个冷启动可选择合并为单工作器或扩展到流水线大小
- **技术基础**：独立的扩缩容决策
- **实验验证**：支持灵活的资源管理
- **对比分析**：现有方法缺乏弹性

### 局限性分析

#### 局限1：仅评估小模型
- **描述**：仅在 7B 模型上评估
- **表现**：未验证更大模型（如 70B、175B）
- **原因**：实验资源限制
- **影响**：可能不适用于所有模型规模
- **可能的解决方案**：扩展到更大模型

#### 局限2：特定硬件配置
- **描述**：在 A10 GPU 和 16Gbps 网络上评估
- **表现**：未验证其他硬件配置
- **原因**：实验环境限制
- **影响**：可能不适用于所有部署环境
- **可能的解决方案**：扩展到更多硬件配置

#### 局限3：流水线并行的限制
- **描述**：最大流水线并行度限制为 4
- **表现**：边际改善递减
- **原因**：其他冷启动阶段也有开销
- **影响**：进一步优化需要减少其他阶段的开销
- **可能的解决方案**：优化容器创建和运行时初始化

#### 局限4：模型权重格式依赖
- **描述**：依赖 SafeTensors 格式支持流式访问
- **表现**：需要模型使用特定格式
- **原因**：共享内存需要流式读取支持
- **影响**：可能需要格式转换
- **可能的解决方案**：支持更多模型格式
