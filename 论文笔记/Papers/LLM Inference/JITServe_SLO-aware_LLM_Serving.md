## 摘要翻译

### 英文摘要
The integration of Large Language Models (LLMs) into applications ranging from interactive chatbots to multi-agent systems has introduced a wide spectrum of service-level objectives (SLOs) for responsiveness. These include latency-sensitive requests emphasizing per-token latency in streaming chat, deadline-sensitive requests requiring rapid full responses to trigger external tools, and compound requests with evolving dependencies across multiple LLM calls. Despite--or perhaps, because of--this workload diversity and unpredictable request information (e.g., response lengths and dependencies), existing request schedulers have focused on aggregate performance, unable to ensure application-level SLO needs.

This paper presents JITServe, the first SLO-aware LLM serving system designed to maximize service goodput across diverse workloads. JITServe novelly schedules requests using imprecise request information and gradually relaxes this conservatism by refining request information estimates as generation progresses. It applies a grouped margin goodput maximization algorithm to allocate just enough serving bandwidth to satisfy each request's SLO just-in-time (JIT), maximizing residual capacity for others, while deciding the composition of requests in a batch to maximize efficiency and goodput with provable guarantees. Our evaluation across diverse realistic workloads, including chat, deep research, and agentic pipelines, shows that JITServe improves service goodput by 1.4-6.3x, alternatively achieving 28.5%-83.2% resource savings, compared to state-of-the-art designs.

### 中文翻译
大语言模型（LLM）集成到从交互式聊天机器人到多智能体系统的各种应用中，引入了广泛的服务级别目标（SLO）需求。这些包括：在流式聊天中强调每 token 延迟的延迟敏感请求、需要快速完整响应以触发外部工具的截止时间敏感请求、以及跨多个 LLM 调用具有演化依赖关系的复合请求。尽管（或正因为）这种工作负载多样性和不可预测的请求信息（如响应长度和依赖关系），现有的请求调度器专注于聚合性能，无法确保应用级别的 SLO 需求。

本文提出 JITServe，第一个 SLO 感知的 LLM 服务系统，旨在最大化跨多样化工作负载的服务有效吞吐量（goodput）。JITServe 创新性地使用不精确的请求信息进行调度，并随着生成过程的推进逐步放宽这种保守性，通过精细化请求信息估计。它应用分组边际有效吞吐量最大化算法（GMAX），在每个请求的 SLO 上及时（JIT）分配刚好足够的服务带宽，同时最大化其他请求的剩余容量，同时决定批次中请求的组成以最大化效率和有效吞吐量，并提供可证明的保证。我们在多种真实工作负载（包括聊天、深度研究和智能体流水线）上的评估表明，与最先进的设计相比，JITServe 将服务有效吞吐量提高了 1.4-6.3 倍，或者实现 28.5%-83.2% 的资源节省。

### 核心要点提炼
- **研究背景**：LLM 应用场景多样化，引入了三类不同的 SLO 需求（延迟敏感、截止时间敏感、复合请求）
- **研究动机**：现有调度器优化聚合性能，无法满足多样化的应用级 SLO；请求信息（响应长度、依赖关系）天然不精确
- **核心方法**：JIT 调度原则 + 分组边际有效吞吐量最大化算法（GMAX）+ QRF 长度上界估计 + 模式图匹配
- **主要结果**：服务有效吞吐量提升 1.4-6.3 倍，或 28.5%-83.2% 资源节省，接近 Oracle 性能
- **研究意义**：首个 SLO 感知的 LLM 服务系统，首次系统性地处理请求不确定性

## 研究背景与动机

### 领域现状
LLM 已成为从聊天机器人到智能体系统的核心组件。随着 LLM 应用的扩展，用户和开发者对响应性提出了多样化的 SLO 需求。作者通过分析数百万真实 LLM 请求、550+ 用户调查和与服务提供商的深入讨论，发现了三类主导的请求模式。

### LLM 请求的三种模式

#### 1. 延迟敏感请求（Latency-sensitive Requests）
- **特征**：响应以流式方式消费
- **SLO 指标**：TTFT（首 token 延迟）和 TBT（token 间延迟）
- **典型应用**：ChatGPT 风格聊天、AI 客户支持、实时语音转文本
- **用户偏好**：38.1% 的代码生成用户偏好流式交付

#### 2. 截止时间敏感请求（Deadline-sensitive Requests）
- **特征**：需要在指定截止时间前返回完整响应
- **SLO 指标**：E2EL（端到端延迟）
- **典型应用**：云 AIOps、批处理 API、代理交互触发外部工具
- **用户偏好**：30.5% 的代码生成用户偏好快速完整响应

#### 3. 复合请求（Compound Requests）
- **特征**：包含多个依赖的 LLM 调用，形成图结构依赖
- **SLO 指标**：整体 E2EL 截止时间
- **典型应用**：多智能体工作流、深度研究、分层推理
- **挑战**：依赖结构可能动态演化

### 现有方法的局限性

#### 1. 聚合性能优化的局限
- **代表系统**：Autellix（PLAS 调度）、vLLM（FCFS 调度）
- **问题**：优化平均 E2EL 可以导致任意差的有效吞吐量
- **证据**：Autellix 的 SLO 违规率超过 90%

#### 2. 仅针对延迟敏感请求的优化
- **代表系统**：Sarathi-Serve
- **问题**：对截止时间敏感请求表现差，导致高 SLO 违规率
- **原因**：TTFT 和 TBT 主要依赖已知输入长度，但 E2EL 依赖未知输出长度

#### 3. 请求不确定性的挑战
- **响应长度预测困难**：即使使用微调的 BERT 或 Llama3 模型，预测误差仍然很大
- **依赖结构不确定**：复合请求的依赖可能动态演化
- **到达模式波动**：负载在分钟内可变化 5 倍

### 研究动机

#### 关键观察：精确信息 vs 无信息的中间地带
- LLM 响应是自回归生成的，经过数百个解码迭代
- 可以在生成过程中逐步精细化估计
- 初始使用保守的上界估计，随着生成进展逐步放宽

#### 设计目标
1. **通用性**：支持多样化的请求类型和 SLO 需求
2. **有效吞吐量效率**：最大化服务有效吞吐量和资源利用率，提供可证明的保证
3. **可部署性和可扩展性**：与现有服务栈无缝集成

## 研究问题

### 核心研究问题
如何在请求信息不精确的情况下，设计一个 SLO 感知的 LLM 服务系统，以最大化跨多样化工作负载的服务有效吞吐量？

### 子问题
1. 如何在不知道精确响应长度的情况下进行有效的调度？
2. 如何处理复合请求中动态演化的依赖关系？
3. 如何在满足多样化 SLO 的同时最大化批次执行效率？
4. 如何在公平性和效率之间取得平衡？

## 方法概述

### 前置知识

**服务有效吞吐量（Service Goodput）**：满足 SLO 要求的有效 token 数量或请求数量。不同于吞吐量（throughput），goodput 强调的是满足服务质量要求的有效输出。

**SLO（Service Level Objective）**：服务级别目标，定义了请求响应性的具体要求，如 TTFT、TBT、E2EL 等。

**QRF（Quantile Regression Forest）**：分位数回归森林，一种用于预测条件分位数的机器学习方法，可以输出响应长度的上界估计。

### 核心洞察

#### 洞察 1：不精确但可操作的信息
- 虽然精确预测响应长度困难，但估计上界更可靠
- 上界自然转化为最大服务带宽需求的保守估计
- 随着生成进展，可以逐步精细化上界

#### 洞察 2：模式图匹配
- 复合请求具有图结构的执行模式
- 历史请求的模式图可以用于预测新请求的依赖结构
- 增量图匹配可以在信息逐步揭示时进行精细化

#### 洞察 3：边际有效吞吐量优先
- 调度应该优先考虑边际有效吞吐量/带宽比最高的请求
- 这自然偏好高回报相对于带宽需求的请求
- 消除了对时间帧长度的敏感性

### 核心算法

#### 算法总览

**输入**：LLM 推理请求（附带 SLO）、系统状态
**输出**：调度决策、带宽分配策略
**关键组件**：
- Request Analyzer：请求信息估计和精细化
- SLO-Aware Scheduler：SLO 感知调度器
- GMAX Algorithm：分组边际有效吞吐量最大化算法

**阶段1：请求分析**
1. 使用 QRF 估计响应长度上界
2. 计算最小服务带宽需求
3. 通过模式图匹配估计依赖结构
4. 为复合请求分配子截止时间

**阶段2：GMAX 调度**
1. 计算每个请求的优先级（边际有效吞吐量/带宽）
2. 按输入长度排序，使用滑动窗口找最优批次
3. 选择组优先级最高的批次执行
4. 动态调整截断参数 p

**阶段3：在线精细化**
1. 每 50 个 token 重新估计响应长度上界
2. 更新依赖结构估计
3. 调整带宽分配
4. 在必要时进行抢占

#### 详细流程

1. **响应长度上界估计**：
   - 使用 QRF 模型（7ms/次，比 BERT 快 70 倍）
   - 输出高分位数的长度上界
   - 每 50 个 token 重新估计，逐步精细化
   - 避免低估导致 SLO 违规，避免过度估计浪费带宽

2. **依赖结构估计**：
   - 将每个请求表示为模式图（节点=LLM/工具调用，边=依赖）
   - 使用高斯核函数计算节点和边的相似度
   - 增量扩展部分图，剪枝不匹配的历史模式
   - 通过累积份额比例分配子截止时间

3. **GMAX 调度算法**：
   - 计算最小服务带宽：bw(r) = tgen(r) / trem(r)
   - 计算优先级：Priority(r) = goodput(r) / tgen(r)
   - 过滤候选请求（保留优先级 >= p * 最高优先级）
   - 按输入长度排序，滑动窗口找最优批次
   - 选择组优先级最高的批次

4. **抢占机制**：
   - 估计抢占开销：goodput_loss = stall_duration * token_generation_speed
   - 仅当收益超过开销时进行抢占
   - 限制在离散时间帧（50 步，约 300ms）内更新
   - 实际开销 < 1%

### 方法关键创新点

1. **首个 SLO 感知的 LLM 服务系统**：
   - 支持三类请求模式（延迟敏感、截止时间敏感、复合请求）
   - 显式对齐应用级 SLO 需求与底层执行
   - 仅需少量代码修改即可集成到现有服务栈

2. **不精确信息的渐进精细化**：
   - 初始使用保守的上界估计
   - 随着生成进展逐步放宽保守性
   - QRF 轻量级估计（7ms/次）支持在线精细化

3. **GMAX 调度算法**：
   - 优先调度边际有效吞吐量/带宽比最高的请求
   - 分组调度兼顾有效吞吐量和批次效率
   - 可证明的竞争性能保证（1/8.56 近似比）

4. **模式图匹配**：
   - 增量图匹配支持依赖结构的逐步揭示
   - 累积份额比例分配子截止时间
   - 轻量级匹配（<5ms），支持在线执行

## 实验结果

### 实验目标
验证 JITServe 在多样化工作负载和 SLO 需求下的服务有效吞吐量，与最先进设计比较。

### 实验设置

#### 硬件环境
- 16 块 NVIDIA A100 GPU 集群

#### 模型
- Llama-3.1-8B（Dense）
- Qwen2.5-14B（Dense）
- Qwen3-30B-MoE-A3B（MoE）
- Llama-3.1-70B（Dense）

#### 数据集和工作负载
- **Chatbot**：Alpaca + LMsys-chat，延迟敏感 + 复合请求
- **Deep Research**：Search Arena 基准，复合请求
- **Agentic CodeGen**：代码生成工作流，截止时间敏感
- **Math Reasoning**：数学推理任务，复合请求
- 请求到达遵循 Microsoft 真实 LLM 服务跟踪

#### SLO 设置
- 延迟敏感请求：TTFT = 2s，TBT = 100ms
- 截止时间敏感请求：E2EL = 20s
- 复合请求：E2EL = 20s × 阶段数

#### 基线方法
- **vLLM**：FCFS 调度 + 连续批处理 + PagedAttention
- **Sarathi-Serve**：vLLM + 分块 prefill，优化 TTFT/TBT
- **Autellix**：程序级 LAS 调度，优化 E2EL
- **LTR**：基于 LLM 预测的相对长度排序（模仿 SJF）

#### 评估指标
- **Token-level Goodput**：满足 SLO 的 token 数量
- **Request-level Goodput**：满足 SLO 的请求数量
- **TTFT/TBT/E2EL**：传统性能指标
- **SLO Violation Rate**：SLO 违规率

### 方法优势详解

#### 优势1：处理请求不确定性
- **描述**：通过 QRF 上界估计和增量精细化，在不确定信息下做出保守但可调整的调度决策
- **技术基础**：QRF 轻量级（7ms/次），支持在线精细化
- **实验验证**：接近 Oracle 性能（3-9% 差距）
- **对比分析**：BERT/Llama3 预测器经常低估长度，导致 SLO 违规

#### 优势2：支持多样化 SLO
- **描述**：统一处理延迟敏感、截止时间敏感和复合请求
- **技术基础**：模式图匹配 + 子截止时间分配
- **实验验证**：在所有请求类型上均优于基线
- **对比分析**：Sarathi-Serve 仅优化延迟敏感请求，Autellix 仅优化平均 E2EL

#### 优势3：可证明的性能保证
- **描述**：GMAX 算法提供 1/8.56 的竞争性能比
- **技术基础**：边际有效吞吐量优先级 + 分组调度
- **实验验证**：接近 Oracle 性能
- **对比分析**：现有方法缺乏理论保证

#### 优势4：易于部署
- **描述**：基于 vLLM 实现，仅需约 2800 行代码
- **技术基础**：标准 API 扩展（添加 SLO 参数）
- **实验验证**：吞吐量开销 <4%
- **对比分析**：无需修改模型或数据格式

### 局限性分析

#### 局限1：QRF 模型需要训练
- **描述**：需要为每个模型和工作负载训练 QRF 模型
- **表现**：不同模型可能需要不同的 QRF
- **原因**：响应长度分布因模型和工作负载而异
- **影响**：增加部署前的准备工作
- **可能的解决方案**：在线学习或迁移学习

#### 局限2：模式图匹配依赖历史数据
- **描述**：需要足够的历史请求来构建模式图库
- **表现**：新工作负载可能缺乏匹配的历史模式
- **原因**：模式图需要从历史请求中学习
- **影响**：冷启动时性能可能下降
- **可能的解决方案**：使用默认模式或在线学习

#### 局限3：抢占开销
- **描述**：虽然开销 <1%，但仍需管理
- **表现**：KV 缓存交换需要时间
- **原因**：GPU 内存管理的固有限制
- **影响**：在极端情况下可能影响性能
- **可能的解决方案**：更高效的 KV 缓存管理

#### 局限4：评估范围
- **描述**：主要在 A100 GPU 上评估
- **表现**：未验证其他硬件平台
- **原因**：实验资源限制
- **影响**：可能不适用于所有部署环境
- **可能的解决方案**：扩展到更多硬件平台

### 适用性与场景分析

#### 适用场景
1. **多样化 LLM 服务**
   - 适用原因：支持三类请求模式和多样化 SLO
   - 预期效果：有效吞吐量提升 1.4-6.3 倍
   - 注意事项：需要定义合适的 SLO 参数

2. **代理系统**
   - 适用原因：支持复合请求和依赖结构
   - 预期效果：满足外部工具触发的截止时间
   - 注意事项：需要客户端指定 SLO

3. **多模型部署**
   - 适用原因：支持多模型数据并行
   - 预期效果：一致的性能提升
   - 注意事项：需要为每个模型训练 QRF

4. **批处理 API**
   - 适用原因：支持截止时间敏感请求
   - 预期效果：资源节省 28.5%-83.2%
   - 注意事项：需要合理的截止时间设置

#### 不适用场景
1. **单一类型工作负载**
   - 不适用原因：优化收益有限
   - 替代方案：使用 Sarathi-Serve（延迟敏感）或 Autellix（截止时间敏感）

   - 替代方案：使用专用硬件或简化调度

   - 不适用原因：SLO 约束不强
   - 替代方案：使用简单的 FCFS 调度
