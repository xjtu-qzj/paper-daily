## 摘要翻译

### 英文摘要
Mixture-of-Experts (MoE) showcases tremendous potential to scale large language models (LLMs) with enhanced performance and reduced computational complexity. However, its sparsely activated architecture shifts feed-forward networks (FFNs) from being compute-intensive to memory-intensive during inference, leading to substantially lower GPU utilization and increased operational costs. We present MegaScale-Infer, an efficient and cost-effective system for serving large-scale MoE models. MegaScale-Infer disaggregates attention and FFN modules within each model layer, enabling independent scaling, tailored parallelism strategies, and heterogeneous deployment for both modules. To fully exploit disaggregation in the presence of MoE's sparsity, MegaScale-Infer introduces ping-pong pipeline parallelism, which partitions a request batch into micro-batches and shuttles them between attention and FFNs for inference. Combined with distinct model parallelism for each module, MegaScale-Infer effectively hides communication overhead and maximizes GPU utilization. To adapt to disaggregated attention and FFN modules and minimize data transmission overhead (e.g., token dispatch), MegaScale-Infer provides a high-performance M2N communication library that eliminates unnecessary GPU-to-CPU data copies, group initialization overhead, and GPU synchronization. Experimental results indicate that MegaScale-Infer achieves up to 1.90x higher per-GPU throughput than state-of-the-art solutions.

### 中文翻译
混合专家（MoE）在以亚线性计算复杂度扩展大语言模型（LLM）方面展现了巨大潜力。然而，其稀疏激活架构在推理期间将前馈网络（FFN）从计算密集型转变为内存密集型，导致 GPU 利用率大幅降低和运营成本增加。我们提出 MegaScale-Infer，一个高效且经济的 MoE 模型服务系统。MegaScale-Infer 在每个模型层内解耦注意力和 FFN 模块，实现独立扩展、定制化并行策略和异构部署。为了在 MoE 稀疏性存在下充分利用解耦优势，MegaScale-Infer 引入乒乓流水线并行，将请求批次分成微批次，在注意力和 FFN 之间穿梭进行推理。结合每个模块的不同模型并行策略，MegaScale-Infer 有效隐藏通信开销并最大化 GPU 利用率。为了适应解耦的注意力和 FFN 模块并最小化数据传输开销（如 token 分发），MegaScale-Infer 提供高性能 M2N 通信库，消除不必要的 GPU 到 CPU 数据复制、组初始化开销和 GPU 同步。实验结果表明，MegaScale-Infer 实现了比最先进解决方案高达 1.90 倍的每 GPU 吞吐量。

### 核心要点提炼
- **研究背景**：MoE 模型的稀疏激活导致 FFN 在解码阶段 GPU 利用率低（如 Mixtral 8x22B 仅 25% MFU）
- **研究动机**：现有系统将注意力和 FFN 作为整体服务，无法充分利用 MoE 的稀疏性
- **核心方法**：解耦注意力和 FFN + 乒乓流水线并行 + M2N 通信库 + 异构部署
- **主要结果**：每 GPU 解码吞吐量提升 1.90 倍，异构部署下每成本吞吐量提升 1.86 倍
- **研究意义**：首个针对大规模 MoE 推理的解耦专家并行系统

## 研究背景与动机

### 领域现状
LLM 的有效性越来越依赖于模型参数的扩展。由于稀疏激活架构，MoE 模型成为实际选择：它动态将输入 token 路由到前馈网络（FFN）的子集（专家），而不是激活所有参数，实现亚线性计算复杂度扩展。

### MoE 推理的独特挑战

#### GPU 利用率问题
- **Dense 模型**：FFN 是计算密集型，可通过批处理充分利用 GPU
- **MoE 模型**：稀疏性导致每个专家处理的 token 减少
- **具体例子**：Mixtral 8x22B，batch size=156 时，每个专家仅处理 39 个 token
- **理论 MFU**：FFN 利用率 = topk/#expert = 2/8 = 25%

#### Roofline 模型分析
- A100 GPU：需要 batch size ≥ 156 才能充分利用矩阵乘法能力
- MoE 稀疏性：实际每个专家的 batch size = B × topk / #expert
- 结果：FFN 不再是计算密集型，变成内存密集型

#### 现有方法的局限
1. **vLLM/TensorRT-LLM**：
   - 将模型作为整体服务
   - FFN batch size 小，GPU 利用率低
   - 仅支持张量并行或流水线并行

2. **InfiniteLLM**：
   - 解耦注意力以增加解码 batch size
   - 专注于长上下文场景（内存限制而非稀疏性）
   - 通信模式相对简单，不适用于 MoE

### 研究动机

#### 关键洞察：解耦注意力和 FFN 的优势

1. **独立扩展**：
   - 复制注意力模块以增加 FFN 的 batch size
   - 从多个注意力副本聚合请求
   - FFN 从内存密集型转变为计算密集型

2. **异构部署**：
   - 注意力模块：内存密集型，适合高带宽 GPU
   - FFN 模块：计算密集型，适合高计算力 GPU
   - 可以使用不同类型的 GPU 实现最优成本效益

#### 技术挑战

1. **空闲时间问题**：
   - 解耦导致注意力和 FFN 在对方计算时空闲
   - 等待网络传输时也空闲
   - 需要建立乒乓流水线保持忙碌

2. **M2N 通信问题**：
   - 任意并行配置将 All2All 变为 M2N 通信
   - 现有通信库（NCCL）性能不佳
   - 需要专用通信库

## 研究问题

### 核心研究问题
如何设计一个高效且经济的系统来服务大规模 MoE 模型，充分利用其稀疏激活特性？

### 子问题
1. 如何解耦注意力和 FFN 模块并实现独立扩展？
2. 如何在解耦架构下最大化 GPU 利用率？
3. 如何实现高效的 M2N 通信？
4. 如何支持异构硬件部署？

## 方法概述

### 前置知识

**MoE（Mixture of Experts）**：将 FFN 替换为多个专家网络，门控网络将 token 路由到 top-k 个专家。如 Mixtral 8x22B 有 8 个专家，top-2 选择。

**专家并行（Expert Parallelism）**：每个设备仅包含部分专家，需要 All2All 通信进行 token 路由。

**乒乓流水线并行**：将批次分成微批次，在注意力和 FFN 之间穿梭，隐藏通信开销。

### 核心洞察

#### 洞察 1：解耦的双重优势
- 独立扩展：复制注意力增加 FFN batch size
- 异构部署：不同类型 GPU 适配不同模块

#### 洞察 2：乒乓流水线
- 将批次分成微批次
- 注意力和 FFN 交替处理微批次
- 通信与计算重叠

#### 洞察 3：M2N 通信优化
- NCCL 的 GPU-to-CPU 复制、组初始化、GPU 同步引入开销
- CPU-based RDMA 可以避免这些开销
- 专用通信库可以实现更低延迟

### 核心算法

#### 算法总览

**输入**：MoE 模型、硬件配置、SLO 约束
**输出**：部署计划（并行策略、微批次数量、batch size、硬件配置）
**关键组件**：
- 解耦专家并行：注意力和 FFN 分离部署
- 乒乓流水线并行：微批次穿梭
- M2N 通信库：高性能通信
- 部署计划搜索：优化吞吐量/成本

**阶段1：部署计划搜索**
1. 枚举注意力和专家节点的张量并行配置
2. 计算注意力节点数量以平衡计算时间
3. 搜索最优微批次数量
4. 二分搜索最大 batch size 满足 SLO
5. 选择吞吐量/成本最高的方案

**阶段2：乒乓流水线执行**
1. 将批次分成 m 个微批次
2. 微批次在注意力和 FFN 之间穿梭
3. 通信与计算重叠
4. 最小化 GPU 空闲时间

**阶段3：M2N 通信**
1. 预注册张量缓冲区
2. 使用 RDMA write with immediate 传输
3. 轮询完成队列确认
4. 避免 GPU-to-CPU 复制和 GPU 同步

#### 详细流程

1. **解耦专家并行**：
   - 注意力模块：数据并行复制，存储 KV 缓存
   - FFN 模块：专家并行，每个节点存储一个专家
   - 张量并行：节点内使用 NVLink

2. **乒乓流水线并行**：
   - 将批次分成 m 个微批次
   - 微批次在注意力和 FFN 之间穿梭
   - 每个 MoE 层需要两次通信（A2E 和 E2A）
   - 约束条件：
     - Ta ≈ Te（平衡计算时间）
     - Tc < Tf（隐藏通信）
     - m × Tf ≥ 2 × (Tf + Tc)（填充流水线）

3. **部署计划搜索**：
   - 枚举 tpe（专家 TP）和 tpa（注意力 TP）
   - 计算 na（注意力节点数）以平衡 Ta 和 Te
   - 搜索最优 m（微批次数量，3-4）
   - 二分搜索最大 batch size B
   - 目标：最大化 B/Ttotal / 成本

4. **M2N 通信库**：
   - Sender：CUDA event 等待 → RDMA write → 轮询完成 → 解除阻塞
   - Receiver：CUDA event 等待 → 轮询完成 → GDRCopy flush → 解除阻塞
   - 优化：高优先级 ACK、拥塞控制调优

5. **异构部署**：
   - 注意力节点：H20（高内存带宽、大容量）
   - 专家节点：L40S（高计算力/成本比）
   - 部署计划搜索同时枚举异构配置

6. **负载均衡**：
   - 基于专家热度的设备内冗余
   - 贪心近似算法分配专家
   - 最小化最大计算成本

1. **首个解耦专家并行系统**：

   - 注意力节点使用高带宽 GPU
   - 专家节点使用高计算力 GPU
   - 最大化每成本吞吐量

## 实验结果

### 实验目标
验证 MegaScale-Infer 在大规模 MoE 推理中的效率和成本效益。

### 实验设置

#### 硬件环境
- **同构集群**：8 个节点，每节点 8 块 A100 80GB GPU
  - NVLink 400GB/s，InfiniBand 200Gbps
- **异构集群**：H20 + L40S
  - H20：NVLink 900GB/s，4×400Gbps NIC
  - L40S：PCIe，2×400Gbps NIC

#### 模型
| 模型 | 参数量 | 层数 | 专家数 | top-k |
|------|--------|------|--------|-------|
| Mixtral 8x22B | 141B | 56 | 8 | 2 |
| DBRX | 132B | 40 | 16 | 4 |
| Scaled-MoE | 317B | 48 | 32 | 4 |

#### 基线方法
- **vLLM**：整体服务，FCFS 调度
- **TensorRT-LLM**：整体服务，支持专家并行

#### 评估指标
- **每 GPU 解码吞吐量**：tokens/s/GPU
- **每成本解码吞吐量**：tokens/s/成本单位
- **TBT**：Time-Between-Tokens
- **M2N 通信延迟**：中位数和 P99

### 主要结果

#### 同构部署（A100）

| 模型 | vLLM | TensorRT-LLM | MegaScale-Infer | 提升 |
|------|------|---------------|-----------------|------|
| Mixtral 8x22B | 1x | 2.0x | **2.56x** | 1.28x vs TRT |
| DBRX | 1x | 1.8x | **2.30x** | 1.28x vs TRT |
| Scaled-MoE | 1x | 3.74x | **7.11x** | 1.90x vs TRT |

#### 异构部署（H20 + L40S）

| 模型 | vLLM (H20) | TensorRT-LLM (H20) | MegaScale-Infer | 提升 |
|------|------------|---------------------|-----------------|------|
| Mixtral 8x22B | 1x | 1.74x | **3.24x** | 1.86x vs TRT |
| DBRX | 1x | 1.65x | **2.89x** | 1.75x vs TRT |
| Scaled-MoE | 1x | 2.10x | **3.89x** | 1.85x vs TRT |

#### M2N 通信性能

| 接收方数量 | NCCL 延迟 | M2N 延迟 | 提升 |
|-----------|----------|----------|------|
| 8 | 150μs | 35μs | 4.3x |
| 16 | 250μs | 50μs | 5.0x |
| 32 | 500μs | 80μs | 6.25x |

#### 关键发现
1. 同构部署下，MegaScale-Infer 实现 1.28-1.90 倍吞吐量提升
2. 异构部署下，每成本吞吐量提升 1.75-1.86 倍
3. M2N 通信延迟比 NCCL 低 4.2-6.25 倍
4. TBT 与基线相当（乒乓流水线隐藏通信开销）

### 消融实验

#### 解耦架构的效果
- 解耦注意力和 FFN 显著提高 FFN 的 GPU 利用率
- FFN 从内存密集型转变为计算密集型
- 批处理效率大幅提升

#### 乒乓流水线的效果
- 有效隐藏通信开销
- 微批次数量 3-4 即可满足约束
- 通信/计算比 < 0.5 时效果最佳

#### M2N 通信优化的效果
- 消除 GPU-to-CPU 复制：延迟降低 40%
- 消除组初始化：延迟降低 30%
- 消除 GPU 同步：P99 延迟显著降低

### 方法优势详解

#### 优势1：显著提高 GPU 利用率
- **描述**：通过解耦和独立扩展，FFN 从内存密集型转变为计算密集型
- **技术基础**：复制注意力模块增加 FFN batch size
- **实验验证**：每 GPU 吞吐量提升 1.90 倍
- **对比分析**：现有系统将模型作为整体，无法充分利用稀疏性

#### 优势2：有效隐藏通信开销
- **描述**：乒乓流水线并行将通信与计算重叠
- **技术基础**：微批次穿梭、约束条件设计
- **实验验证**：TBT 与基线相当
- **对比分析**：简单的解耦会导致严重的空闲时间

#### 优势3：高性能通信
- **描述**：M2N 通信库消除 NCCL 的三大开销
- **技术基础**：CPU-based RDMA、流量优化
- **实验验证**：延迟比 NCCL 低 4.2-6.25 倍
- **对比分析**：NCCL 为通用集合通信设计，不适用于 M2N 模式

#### 优势4：支持异构部署
- **描述**：不同模块使用最适合的 GPU 类型
- **技术基础**：部署计划搜索枚举异构配置
- **实验验证**：每成本吞吐量提升 1.86 倍
- **对比分析**：现有系统仅支持同构部署

### 局限性分析

#### 局限1：仅评估解码阶段
- **描述**：主要关注解码效率，prefill 效果有限
- **表现**：端到端吞吐量提升低于解码吞吐量
- **原因**：prefill 是计算密集型，解耦优势有限
- **影响**：可能需要与 prefill/decoding 分离结合
- **可能的解决方案**：与 Splitwise 等技术结合

#### 局限2：乒乓流水线的延迟影响
- **描述**：为了最大化吞吐量，可能增加单个微批次的延迟
- **表现**：TBT 与基线相当，但未显著降低
- **原因**：填充流水线需要多个微批次
- **影响**：对延迟敏感场景可能不理想
- **可能的解决方案**：自适应微批次数量

#### 局限3：通信库的适用范围
- **描述**：M2N 通信库针对特定场景优化
- **表现**：在连接数量进一步增加时可能需要 GPU 并行
- **原因**：CPU 单线程处理能力有限
- **影响**：可能不适用于更大规模部署
- **可能的解决方案**：结合 GPU 并行处理

#### 局限4：负载均衡开销
- **描述**：需要基于热度的专家冗余和调度
- **表现**：需要监控和调整
- **原因**：MoE 的负载不均衡特性
- **影响**：增加系统复杂性
- **可能的解决方案**：更智能的负载均衡算法

### 适用性与场景分析

#### 适用场景
1. **大规模 MoE 模型服务**
   - 适用原因：稀疏性导致 GPU 利用率低
   - 预期效果：吞吐量提升 1.90 倍
   - 注意事项：需要足够的 GPU 资源

2. **异构集群部署**
   - 适用原因：不同模块适合不同 GPU
   - 预期效果：每成本吞吐量提升 1.86 倍
   - 注意事项：需要部署计划搜索

3. **成本敏感场景**
   - 适用原因：显著降低服务成本
   - 预期效果：成本降低 1.5-2.0 倍
   - 注意事项：需要权衡延迟和吞吐量

4. **高吞吐量需求**
   - 适用原因：最大化 GPU 利用率
   - 预期效果：显著提高吞吐量
   - 注意事项：可能增加单请求延迟

#### 不适用场景
1. **Dense 模型**
   - 不适用原因：无稀疏性，解耦优势有限
   - 替代方案：使用 vLLM 等整体服务

2. **极低延迟要求**
   - 不适用原因：乒乓流水线可能增加延迟
   - 替代方案：使用简化调度

3. **小规模部署**
   - 不适用原因：解耦需要额外资源
   - 替代方案：使用整体服务
