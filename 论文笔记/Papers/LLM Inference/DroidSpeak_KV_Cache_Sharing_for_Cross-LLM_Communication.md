## 摘要翻译

### 英文摘要
Compound AI systems, such as agentic systems, are an emerging trend in large-scale enterprise settings, with multiple LLMs specialized for different users, tasks, and/or roles working together. In these scenarios, different models often process inputs that share the same context prefix. Although much work was done in the past to enable the reuse of prefix KV caches across inputs for a single model, how to enable one model to reuse the prefix KV caches of a different model remains an open question.

We introduce DroidSpeak, the first distributed LLM inference system that enables KV cache reuse across distributed nodes running inference of different LLMs, so long as the LLMs have the same architecture. We present the first study that aims at understanding the impact of sharing KV caches across different LLMs, and if/when such sharing affects quality. Inspired by the findings, we present DroidSpeak, which selectively recomputes a few layers of the KV cache produced by another LLM and reuses the remaining layers, with negligible quality loss.

### 中文翻译
复合 AI 系统（如智能体系统）是大规模企业环境中的新兴趋势，多个专门针对不同用户、任务和/或角色的 LLM 协同工作。在这些场景中，不同的模型通常处理共享相同上下文前缀的输入。尽管过去已有大量工作实现了单个模型在不同输入之间复用前缀 KV 缓存，但如何让一个模型复用另一个不同模型的前缀 KV 缓存仍然是一个开放性问题。

我们引入了 DroidSpeak，这是第一个分布式 LLM 推理系统，能够在运行不同 LLM 推理的分布式节点之间实现 KV 缓存复用，只要这些 LLM 具有相同的架构。我们首次研究了跨不同 LLM 共享 KV 缓存的影响，以及这种共享在何时会影响质量。基于这些发现，我们提出了 DroidSpeak，它选择性地重新计算另一个 LLM 产生的 KV 缓存的少数几层，并复用其余层，质量损失可忽略不计。

### 核心要点提炼
- **研究背景**：复合 AI 系统中多个 LLM 需要处理共享的上下文前缀
- **研究动机**：跨不同 LLM 复用 KV 缓存可以显著减少重复计算
- **核心方法**：通过离线 profiling 识别关键层，选择性地重新计算这些层
- **主要结果**：prefill 延迟降低 1.7-3.1x，吞吐量提升最高 4x
- **研究意义**：首次实现跨不同 LLM 的 KV 缓存复用

## 研究背景与动机

### 领域现状
LLM 推理已成为工业界最消耗资源的工作负载之一，需要不断扩大的 GPU 集群。为了减少计算需求，一个常见的优化是让运行相同 LLM 的 GPU 机器通过网络共享复用输入前缀的 KV 缓存。

然而，一个新兴趋势是在一个 GPU 集群中托管多个不同的 LLM。原因是随着 LLM 用于更复杂或个性化的任务，需要多个 LLM（通常从同一个基础模型微调而来）来服务不同的用户、任务或角色。

### 现有方法的局限性
1. **单模型 KV 缓存复用的局限**：
   - 现有技术仅支持同一模型在不同输入之间复用 KV 缓存
   - 无法跨不同模型共享 KV 缓存
   - 直接复用不同模型的 KV 缓存会导致质量大幅下降

2. **多 LLM 服务的挑战**：
   - 多个 LLM 需要处理共享的上下文（如对话历史、文档）
   - 每个模型都需要独立计算 KV 缓存，造成重复计算
   - 长上下文的 prefill 延迟成为性能瓶颈

3. **Prefill 干扰问题**：
   - TTFT 随输入长度超线性增长
   - 长输入将 prefill 延迟变成端到端瓶颈
   - 例如：输入长度增加 4x 可能导致吞吐量下降 32x

### 研究动机
复合 AI 系统中的三个典型用例：

1. **智能体工作流**：
   - 不同的智能体（如编码、测试、调试）使用不同的微调模型
   - 智能体之间需要共享对话历史以保持一致性
   - 例如：测试智能体需要读取编码智能体的对话历史

2. **个性化模型**：
   - 为不同用户或任务定制的个性化助手
   - 这些模型通常处理重叠的上下文（如相同的新闻、文档）
   - 例如：不同用户的助手回答关于同一新闻的查询

3. **多 LLM/多 LoRA 服务**：
   - 模型需要定期更新以融入新信息
   - 多个 LoRA 适配器需要并发服务
   - 例如：ChatGPT API 每两个月发布新版本

## 研究问题

### 核心研究问题
如何在不同 LLM 之间高效复用 KV 缓存，以减少重复计算，同时保持生成质量？

### 子问题
1. 直接复用不同 LLM 的 KV 缓存会对质量产生什么影响？
2. 不同层对 KV 缓存差异的敏感性是否不同？
3. 如何确定哪些层需要重新计算，哪些可以复用？
4. 如何高效地加载和重新计算 KV 缓存？

## 方法概述

### 前置知识

**KV 缓存**：在 Transformer 模型中，每个 attention 层产生 Key (K) 和 Value (V) 张量。所有层的 KV 张量合称为 KV 缓存。

**E 缓存**：Embedding 缓存，是每层的起始点，用于后续的 Transformer 计算。

**微调模型**：从同一基础模型微调而来的模型，共享相似的架构和基础理解能力。

### 核心洞察

#### 洞察 1：直接复用整个 KV 缓存会导致质量大幅下降
- 直接复用 sender 模型的 KV 缓存给 receiver 模型
- 在 HotpotQA 数据集上，大多数模型对的准确率下降超过 50%
- 不同数据集和模型对的质量下降程度不同

#### 洞察 2：只有少量层对 KV 缓存复用敏感
- 不同层对 KV 缓存差异的敏感性不同
- 平均只有约 11% 的层是"关键层"
- 这些关键层的 F1 分数下降超过 10%

#### 洞察 3：关键层的敏感性在不同输入之间相对稳定
- 关键层的身份在不同输入之间变化很小
- 这使得基于示例输入进行 profiling 成为可能
- 非关键层的方差不显著

### 核心算法

#### 算法总览

**输入**：sender 模型和 receiver 模型、训练数据集、延迟 SLO
**输出**：重新计算配置（哪些层需要重新计算）、KV 缓存加载策略
**关键超参数**：
- 重新计算的层数：在 Pareto 前沿上选择
- 精度阈值：质量下降在 5% 以内
- Profiling 粒度：2 层一组

**阶段1：离线 Profiling**
1. 使用训练数据集评估每层的敏感性
2. 识别关键层（F1 分数下降超过 10%）
3. 构建 Pareto 前沿：在每个重新计算层数下达到最高 F1 分数
4. 选择满足精度要求的最小重新计算配置

**阶段2：在线推理**
1. 根据当前工作负载选择重新计算配置
2. 从 sender 模型加载 E 缓存（在 transition 层）
3. 选择性地重新计算关键层
4. 复用其他层的 KV 缓存
5. 流水线化加载和重新计算

#### 详细流程

1. **离线 Profiling**：
   - 对每个模型对，使用训练数据集（如 HotpotQA 的 50 个上下文）
   - 评估每层单独复用时的质量影响
   - 识别关键层（F1 分数下降超过 10%）
   - 构建 Pareto 前沿：
     - X 轴：重新计算的层数
     - Y 轴：F1 分数
     - 选择满足精度要求的最小重新计算配置

2. **重新计算配置选择**：
   - 根据延迟 SLO 动态选择配置
   - 高工作负载时：选择最小重新计算层数（最大化复用）
   - 低工作负载时：选择满足 SLO 的最高精度配置

3. **Smart KV Cache Loading**：
   - 问题：非连续的关键层会导致多个 transition 点
   - 解决方案：重新计算连续的层组，包括关键层之间的非关键层
   - 原因：每次加载 E 缓存都会引入误差，多个 transition 点会累积误差

4. **流水线化加载和重新计算**：
   - 首先传输 transition 层的 E 缓存
   - 立即开始重新计算后续层
   - 同时并行加载复用层的 KV 缓存
   - 隐藏传输延迟

### 方法关键创新点

1. **首次实现跨不同 LLM 的 KV 缓存复用**：
   - 之前的工作仅支持同一模型的 KV 缓存复用
   - DroidSpeak 首次研究并实现了跨不同模型的 KV 缓存复用
   - 为复合 AI 系统提供了新的优化方向

2. **发现层-wise 敏感性模式**：
   - 只有约 11% 的层是关键层
   - 关键层的身份在不同输入之间相对稳定
   - 这使得基于 profiling 的优化成为可能

3. **连续层组重新计算策略**：
   - 选择重新计算连续的层组，而非分散的关键层
   - 减少 transition 点数量，降低误差累积
   - 在效率和质量之间取得更好的平衡

4. **Smart KV Cache Loading**：
   - 流水线化 E 缓存加载和层重新计算
   - 并行传输和计算，隐藏网络延迟
   - 显著减少 TTFT

### 实验设置

#### 基线方法
- **Full Prefill**：receiver 模型完整预填充文本上下文（最高质量，最高延迟）
- **Full KV Cache Reuse**：直接复用 sender 模型的整个 KV 缓存（最低延迟，质量下降）
- **CacheBlend**：基于注意力分数选择重要 token 重新计算

#### 评估指标
- **F1 分数**：问答任务（HotpotQA, 2WikiMultiQA, MultiFieldQA）
- **Rouge-L 分数**：摘要任务（Multi-News）
- **代码相似度**：代码补全任务（LCC, RepoBench-P）
- **TTFT**：Time-to-First-Token
- **TBT**：Time-Between-Tokens
- **E2E**：端到端延迟

#### 实验环境
- 2 个 A100 虚拟机（Microsoft Azure）
- 每个虚拟机 8 块 80GB A100 GPU
- InfiniBand 连接
- 集成 vLLM Production Stack 和 LMCache

### 主要结果

#### Prefill 延迟减少

| 方法 | 平均延迟减少 | 范围 |
|------|-------------|------|
| Full KV Reuse | 3.5x | 2.5-4x |
| CacheBlend | 2.2x | 1.8-2.8x |
| **DroidSpeak** | **2.1x** | **1.7-3.1x** |

#### 质量保持

| 方法 | 平均质量下降 | 范围 |
|------|-------------|------|
| Full KV Reuse | 35% | 20-50% |
| CacheBlend | 8% | 5-15% |
| **DroidSpeak** | **<1%** | **0-2%** |

#### 吞吐量提升

| 模型对 | 吞吐量提升 |
|--------|-----------|
| phi-adapter vs phi-task | 4x |
| mistrallite vs mistral-7b | 3.5x |
| llama-3.1-70b-instruct vs llama-3.1-70b | 3x |
| llama-3-70b-instruct vs llama-3-70b | 3x |

#### 关键发现
1. DroidSpeak 在保持质量的同时，将 prefill 延迟降低 1.7-3.1x
2. 相比 CacheBlend，DroidSpeak 的质量高 5-33%（平均 16%）
3. 在线服务场景下，吞吐量提升最高 4x
4. Profiling 结果在不同数据集和模型类型上具有鲁棒性

### 消融实验

#### 关键层分析
- 平均只有 11% 的层是关键层
- 关键层的身份在不同输入之间相对稳定
- 重新计算连续层组比重新计算分散的关键层效果更好

#### Profiling 粒度
- 2 层一组的 profiling 粒度将 profiling 时间减少约 3x
- 对最终质量影响可忽略

#### 自适应配置选择
- 根据工作负载动态选择重新计算配置
- 高负载时：最大化复用，最小化延迟
- 低负载时：最大化精度

### 方法优势详解

#### 优势1：显著减少重复计算
- **描述**：通过复用 sender 模型的 KV 缓存，避免 receiver 模型的完整 prefill
- **技术基础**：发现只有 11% 的层是关键层，需要重新计算
- **实验验证**：在 8 个模型对上，平均 prefill 延迟降低 2.1x
- **对比分析**：相比完整 prefill，计算量减少约 80%

#### 优势2：保持高质量
- **描述**：选择性地重新计算关键层，保持生成质量
- **技术基础**：基于 profiling 识别关键层，重新计算连续层组
- **实验验证**：质量下降在 0-2% 以内，远优于直接复用
- **对比分析**：相比 CacheBlend，质量高 5-33%

#### 优势3：与现有系统兼容
- **描述**：已集成到 vLLM 和 LMCache 中
- **技术基础**：使用标准接口（store、fetch、partial_prefill）
- **实验验证**：在 Kubernetes 集群上验证了在线服务性能
- **对比分析**：无需修改模型架构，易于部署

### 局限性分析

#### 局限1：需要离线 Profiling
- **描述**：每个模型对需要进行离线 profiling 以识别关键层
- **表现**：Llama-3-8B（32 层）需要 3 小时 profiling
- **原因**：需要评估每层的敏感性，复杂度 O(l²)
- **影响**：增加部署前的准备工作
- **可能的解决方案**：使用分组 profiling 减少时间；利用已有 profiling 结果

#### 局限2：需要存储 E 缓存
- **描述**：transition 层的 E 缓存需要存储和传输
- **表现**：E 缓存大小可达 KV 缓存的 2-4 倍
- **原因**：需要从 transition 层开始重新计算
- **影响**：增加存储和传输开销
- **可能的解决方案**：优化 transition 层的选择；使用压缩技术

#### 局限3：仅适用于相同架构的模型
- **描述**：sender 和 receiver 模型需要有相同的架构
- **表现**：无法跨不同架构的模型共享 KV 缓存
- **原因**：不同架构的 KV 缓存格式不兼容
- **影响**：限制了应用场景
- **可能的解决方案**：研究 KV 缓存的跨架构转换

#### 局限4：质量-延迟权衡的局限性
- **描述**：在某些模型对上，质量-延迟权衡可能不够理想
- **表现**：某些关键层可能需要重新计算较多层
- **原因**：关键层的分布可能不连续
- **影响**：可能无法达到理想的延迟目标
- **可能的解决方案**：研究更精细的重新计算策略

### 适用性与场景分析

#### 适用场景
1. **智能体工作流**
   - 适用原因：不同智能体需要共享对话历史
   - 预期效果：减少智能体之间的通信延迟
   - 注意事项：需要确保智能体使用相同架构的模型

2. **多 LoRA 服务**
   - 适用原因：多个 LoRA 适配器共享基础模型
   - 预期效果：减少并发服务的计算开销
   - 注意事项：需要为每个 LoRA 适配器 profiling

3. **模型更新和迭代**
   - 适用原因：新版本模型可以复用旧版本的 KV 缓存
   - 预期效果：加速模型部署和 A/B 测试
   - 注意事项：需要确保新旧版本架构相同

4. **个性化助手系统**
   - 适用原因：不同用户的个性化模型共享公共上下文
   - 预期效果：减少个性化服务的计算成本
   - 注意事项：需要为每个个性化模型 profiling

#### 不适用场景
1. **不同架构的模型**
   - 不适用原因：KV 缓存格式不兼容
   - 替代方案：使用文本传输或模型转换

2. **实时变化的上下文**
   - 不适用原因：KV 缓存无法复用
   - 替代方案：使用完整 prefill 或优化上下文管理

3. **极短上下文**
   - 不适用原因：prefill 延迟本身很低
   - 替代方案：使用完整 prefill
